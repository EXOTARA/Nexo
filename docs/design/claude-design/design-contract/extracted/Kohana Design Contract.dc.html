01 · Identidad

Sakura Premium

Kohana mantiene su lenguaje visual propio —Sakura Fluent con motivo de pétalos— para nunca parecer un clon. La identidad se percibe como serena, precisa y viva; premium en el detalle, nunca recargada. El principio rector documentado en VISUAL_FOUNDATION_V3.md se eleva a contrato.

Un movimiento dominante
Cada interacción tiene una sola animación protagonista. Nada compite por la atención.
Opacidad + traslación
La materia del movimiento. Nunca se escala texto para animarlo.
Pétalo como firma
El motivo sakura aparece con mesura: marca, estados de escucha y acentos. No decoración de relleno.
Confianza visible
Local vs. nube, escucha activa y acciones aprobables siempre se ven. La transparencia es parte de la marca.

05 · Iconos

Iconografía de Figma

Fuente única: Vector Icon Set for Kohana (Figma). No se mezclan otras familias ni emoji como iconos de UI.

Trazo, no relleno
Grosor 1.5px a 24px, tapas y uniones redondeadas. Relleno solo para estado activo.
Rejilla 24px
Caja óptica de 24 con padding de 2. Tamaños en uso: 16 · 20 · 24.
Color heredado
currentColor por defecto. Los semánticos solo para su significado.
Al exportar de Figma, conservar el nombre data-component de cada icono verbatim para trazabilidad. Pendiente: exportar el set a SVG optimizado cuando la arquitectura se estabilice.
06 · A11y

Reglas de accesibilidad

Contraste AA
Texto ≥ 4.5:1, UI/iconos ≥ 3:1. Sumi sobre paper cumple con holgura; kurenai solo sobre fondos claros.
Foco siempre visible
Anillo de 2px kurenai con offset. Nunca outline:none sin reemplazo.
Objetivos ≥ 32px
Mínimo de clic en escritorio 32px; 40px en controles primarios y de voz.
Teclado total
Todo alcanzable sin ratón; orden de tabulación lógico; atajos anunciados vía UI Automation.
Nunca solo color
Local/nube/bloqueado llevan icono + texto además del color semántico.
Respeta el sistema
Honra escalado de fuente de Windows, alto contraste y reduce motion.
07 · Movimiento

Reglas de animación

Curva de resorte
Overshoot sutil, no solo ease-in-out. Es la diferencia entre "vivo" y "PowerPoint". WPF: EasingFunction personalizada; en WinUI 3, animaciones de resorte nativas.
Un movimiento dominante
Una animación protagonista por interacción. Opacidad + traslación. Nunca escalar texto.
Duración
Micro 120–160ms · transición 200–280ms · entrada de superficie 300–360ms. Nada por encima de 400ms.
Cede ante el Governor
En modo Busy/Game o batería baja, las animaciones se reducen o desactivan. El rendimiento gana siempre (prioridad #1).
El pétalo de escucha es la única animación en bucle permitida, y solo mientras Kohana escucha activamente.
08 · Superficies

Principios del Hub, Peek y Capsule

Tres superficies, una jerarquía de compromiso. Cada una tiene un rol único; no se solapan funciones.

Hub
La ventana completa. Conversación, memoria, skills, ajustes. Máxima densidad y control. Vive con barra lateral (§09).
Peek
Panel flotante, efímero, invocado por voz o atajo. Una respuesta o acción rápida y se va. No es una ventana persistente.
Capsule
Píldora siempre presente, mínima, sobre el escritorio. Estado de escucha y punto de entrada. Cero fricción, casi cero píxeles.
09 · Navegación

Comportamiento de la barra lateral

Dos estados
Expandida (icono + etiqueta, ~240px) y colapsada (solo icono, ~56px). Nunca oculta del todo en el Hub.
Colapso recordado
La preferencia persiste entre sesiones. Auto-colapsa bajo cierto ancho de ventana.
Un ítem activo
Indicador de pétalo/barra kurenai a la izquierda del ítem activo. Solo uno a la vez.
Colapsada = tooltip
En estado icono, cada ítem muestra su etiqueta como tooltip accesible al enfocar/hover.
10 · Plataforma

Restricciones de WPF y Windows

.NET 10 · WPF
Con interop WinForms solo para el icono de bandeja. El diseño se construye con controles WPF reales, no HTML embebido.
Fuentes de sistema
Segoe UI Variable. Sin fuentes web ni descargas para la app.
DPI y escalado
Layout independiente de resolución (per-monitor DPI aware). Nada de posiciones por píxel absoluto.
Pantalla completa
Detecta fullscreen (juegos/video) y no interrumpe. Capsule y notificaciones ceden.
Resource Governor
Umbrales GPU 88% · CPU 92% · RAM 92%. La UI degrada animación y efectos sin matar comandos locales.
Tema del SO
Respeta claro/oscuro y color de acento de Windows donde tenga sentido, sin romper la identidad Sakura.
11 · Flujos

Los tres flujos críticos del lanzamiento

Estos definen el v1. Todo diseño de pantalla se justifica contra ellos.

1
Conversación de voz natural
Wake word → escucha (VAD) → respuesta hablada, con barge-in: el usuario interrumpe y Kohana calla al instante. Debe sentirse instantáneo (streaming de audio por oración), no "graba todo, luego reproduce".
Capsule escuchandoPétalo en bucleCorte inmediato
2
Vision bajo demanda, con transparencia
"¿Qué dice este error?" → OCR/UI Automation local primero → responde mostrando qué detectó antes de dar la respuesta. Nunca guarda en disco; bloquea gestores de contraseñas por proceso y título de ventana.
● LocalVista previa de lo detectadoEn memoria, nunca disco
3
Acción aprobable + transparencia de red
Antes de ejecutar, muestra el plan con nivel Bloqueado / Preguntar / Permitido, registra cada acción, y distingue con claridad cuándo algo sale a la nube vs. cuándo todo quedó local.
Aprobar / rechazar● Sale a la nubeRegistro de acciones
12 · Nunca usar

Elementos visuales prohibidos

✕Gradientes agresivos como fondo
✕Emoji como iconos de interfaz
✕Tarjeta redondeada con borde-acento a la izquierda
✕Escalar texto para animarlo
✕Curvas ease-in-out planas tipo PowerPoint
✕Más de un movimiento simultáneo por interacción
✕Más de dos fondos por superficie
✕Estética de clon (parecerse a OpenClaw u otros)
✕UI de puentes de mensajería (WhatsApp/Telegram/Discord)
✕Semánticos (local/nube/bloqueado) usados como color decorativo
